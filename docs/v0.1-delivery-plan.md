# Icoo v0.1 交付规划

本文档用于把当前 `icoo_lang` 的状态从“持续演进中”收束为一个可以对外说明、对内验收、对重构负责的 `v0.1` 版本。

它关注的不是“还能继续加什么”，而是：

- 哪些能力要在 `v0.1` 冻结下来
- 哪些重构成果必须真正落地
- 交付前必须满足哪些验收条件
- 哪些风险如果不收口，会直接削弱这次版本的可信度

相关文档：

- 当前状态：`docs/mvp-status.md`
- Runtime API：`docs/api.md`
- 语言说明：`docs/language-design.md`
- proxy 重构反馈：`docs/proxy_refactor_feedback.md`

## 一、版本定位

`v0.1` 不是“第一个能跑的版本”。

当前仓库已经明显超过 MVP 阶段，具备完整执行主链、CLI 工具链、模块系统和较大规模标准库。因此 `v0.1` 更准确的定位应该是：

- 第一个可对外说明能力边界的版本
- 第一个以 examples 作为真实回归样例的版本
- 第一个把 `examples/proxy` 重构反馈固化进标准库设计的版本
- 第一个以“稳定性和一致性”而不是“特性数量”作为主目标的版本

一句话概括：

> `v0.1` 要交付的是“可验证、可说明、可继续迭代的基础平台”，不是“特性继续快速膨胀的开发快照”。

## 二、v0.1 的核心目标

### 0. 双交付结果明确

`v0.1` 不是单一交付。

这次版本必须同时保证两个结果成立：

1. `icoo` 本体可交付
2. `examples/proxy` 可作为可用服务样例交付

两者关系如下：

- `icoo` 交付解决“语言平台是否稳定、可说明、可验证”
- `proxy` 交付解决“这轮重构反哺是否真的落到了一个可运行、可验证、可维护的服务样例上”

如果只有前者，没有后者，那么这轮 proxy 反哺价值没有真正落地。
如果只有后者，没有前者，那么 proxy 只是一次局部示例改造，不构成语言版本交付。

### 1. 主链稳定

保证以下主链在文档、CLI、测试和 examples 中都一致成立：

```text
source -> lexer -> parser -> sema -> compiler -> VM -> CLI/API
```

### 2. proxy 重构成果落地

把 `examples/proxy` 当作本次版本最关键的服务端回归样例，确保这轮从 `proxy/icoo_server` 反哺出来的能力不只存在于实现层，而是已经被 example 真正使用：

- `std.object`
- `req.header(name)` / `req.hasHeader(name)` / `req.json`
- 双参 handler `fn(req, res)`
- `res.status(...)` / `res.setHeader(...)` / `res.write(...)` / `res.json(...)` / `res.end(...)`
- `res.proxy(req, options)`
- `req.requestId`
- `std.observe.recent(limit)`
- `std.service.create(...)`
- `std.orm.model(...)`

### 3. 交付口径统一

发版时必须能明确回答以下问题：

- Icoo 当前适合做什么
- 还不适合做什么
- 服务端脚本能力现在走到了哪一步
- 哪些复杂度仍应交给 Go 宿主层
- 用户如何通过 example 和测试快速验证这些能力

### 4. 回归可执行

`v0.1` 不能依赖“人工记得跑哪些测试”。必须把最关键的验证面固定成可重复执行的清单。

## 三、v0.1 冻结范围

### P0：必须冻结并交付

- CLI 主链：`check` / `run` / `bundle` / `build` / `extract` / `inspect`
- `pkg/api` 公开运行时入口
- 文件模块与标准库模块加载
- 当前语言主能力：
  - 函数、闭包、控制流
  - import/export
  - class / inheritance / decorator
  - throw / try / catch / finally / `expr?`
  - chan / go / select
  - type / interface / satisfies
- 当前核心标准库：
  - `std.io`
  - `std.time`
  - `std.math`
  - `std.object`
  - `std.observe`
  - `std.service`
  - `std.db`
  - `std.orm`
  - `std.http.client`
  - `std.http.server`
  - `std.express`
  - `std.json`
  - `std.fs`
- `examples/proxy` 作为服务端样例闭环
- examples 冒烟回归和 `go test ./...` 回归

### P1：允许保留，但不应继续大扩张

- 继续补 tests
- 补文档缺口
- 补 proxy 所需的薄层标准库能力
- 收敛 API 命名和错误风格

### 明确不纳入 v0.1 的内容

为了保证交付，不把下面这些内容塞进 `v0.1`：

- 字节码级 bundle 格式升级
- 完整服务框架化设计
- 重量级 metrics / tracing 系统
- 高性能流式协议桥接框架
- 大规模语法扩展
- 为了追求“纯脚本化”而把本应留在 Go 的工程复杂度迁回语言层

## 四、v0.1 的交付件

发版时至少应包含以下交付件：

### A. `icoo` 交付件

1. 可运行的 CLI
2. 稳定的 `pkg/api`
3. 当前文档集
4. 可批量执行的 examples
5. 能通过的测试套件
6. 一份面向版本的发布说明

### B. `proxy` 可用交付件

1. `examples/proxy` 项目结构完整
2. `app.ic` / `src/main.ic` 入口明确
3. `smoke.ic` 可作为可执行验收脚本
4. 管理接口、健康检查、请求监控链路可用
5. 有独立的运行说明与验收口径

其中 `proxy` 交付尤其关键，因为它承接的是这轮“重构反哺语言”的实际成果。

## 五、必须完成的验收项

### A. `icoo` 交付验收

- `icoo check <file|dir>` 可稳定工作
- `icoo run <file|dir>` 可稳定工作
- `bundle/build/extract/inspect` 主链可用
- `pkg/api` 中已公开入口可用
- `examples/proxy/smoke.ic` 通过
- `std.service`、`std.observe`、`std.orm` 相关能力已被测试覆盖

### B. `proxy` 可用交付验收

- `examples/proxy/app.ic` 可以直接作为项目入口运行
- `examples/proxy/src/main.ic` 暴露 `start` / `stop` / `main`
- `examples/proxy/smoke.ic` 可稳定通过
- proxy 成功路径可完成路由改写、鉴权、上游转发、请求 ID 透传
- `/healthz`、`/readyz`、`/admin/routes`、`/admin/requests`、`/admin/monitor` 可用
- recent requests、counters、latency 监控数据可读
- 不再依赖旧的 store 监控路径作为主路径

### C. 文档验收

- `docs/mvp-status.md` 能准确反映当前状态
- `docs/api.md` 与当前 API 一致
- `docs/proxy_refactor_feedback.md` 能说明为什么 proxy 重构会反哺语言
- `examples/proxy/README.md` 能说明如何运行和验收 proxy
- `v0.1` 发布说明能明确能力边界和限制

### D. 稳定性验收

- `GOFLAGS=-p=1; go test ./...` 通过
- `GOFLAGS=-p=1; go test ./pkg/api -run 'StdService|ExamplesRun/smoke' -v` 通过
- `go run ./cmd/icoo run examples/proxy/smoke.ic` 通过
- `examples` 不出现明显失效或与实现漂移
- parser 错误恢复不再出现容易触发 OOM 的已知路径
- proxy 成功转发路径不再强依赖全量内存缓冲

### E. 一致性验收

- 标准库命名风格一致
- 同类 API 返回形态尽量一致
- 服务端 example 不再保留大面积“旧写法”和“新写法”并存
- 文档中的模块名、函数名、测试入口与仓库实际一致

## 六、交付前的硬性清单

发版前建议按下面顺序走一遍：

1. 合并并复核这轮 proxy 反哺改动
2. 运行低并发全量测试：`GOFLAGS=-p=1; go test ./...`
3. 单独运行 proxy 相关目标测试
4. 检查 `examples/proxy` 是否仍有旧 store/旧监控路径残留
5. 补齐 `std.object` / `std.observe` / `std.service` / `std.orm` 文档说明
6. 补齐 proxy 独立运行说明和验收说明
7. 生成 `v0.1` 发布说明
8. 冻结新增特性，只接受：
   - blocking bug 修复
   - 文档修复
   - 回归测试补强

## 七、建议的发布节奏

### 阶段 1：冻结范围

目标：

- 停止继续扩张语法和大能力面
- 确认 `v0.1` 只做落地和收口

完成标志：

- 本文档确认
- 当前变更范围明确

### 阶段 2：收口实现

目标：

- 把 proxy 反哺出来的能力彻底并入标准库和 example
- 清理示例中的旧路径和重复样板

完成标志：

- proxy example 只走当前推荐路径
- 相关测试稳定通过

### 阶段 3：补足验证

目标：

- 用测试把当前能力钉牢
- 重点补服务端和高级特性交互回归

完成标志：

- 全量测试通过
- 关键 example 冒烟通过

### 阶段 4：文档与发布说明

目标：

- 把“做了什么”和“没做什么”讲清楚

完成标志：

- 发布文档可直接给外部使用者阅读

## 八、v0.1 的风险清单

### 1. 范围继续膨胀

这是当前最大的交付风险。

如果在 `v0.1` 阶段继续不断加入新语法、新协议能力或更重的服务框架抽象，版本会再次从“可交付”滑回“持续开发快照”。

### 2. 文档再次落后于实现

当前能力已经很多。如果不把这次版本文档当成正式交付件，外部理解成本会迅速失控。

### 3. example 失去代表性

如果 `examples/proxy` 仍然保留过多历史路径、临时兼容层或局部重复实现，那么这轮“反哺语言”的成果就没有真正落地。

### 4. 测试通过但交付口径不清

仅有 `go test` 通过还不够。`v0.1` 需要同时回答“边界是什么”，否则使用者会误判它的定位。

### 5. 资源与内存问题被回归引入

已有经验表明：

- parser 错误恢复路径可能造成资源问题
- 并发测试在某些 Windows 环境下可能放大资源压力
- 代理路径如果回退到全量内存缓冲，也会重新引入服务端样例的不可靠性

因此 `GOFLAGS=-p=1` 应当作为发版前默认验证方式之一。

## 九、建议的 v0.1 退出标准

当下面条件同时成立时，可以认为 `v0.1` 具备交付条件：

1. proxy 反哺能力已经实际进入标准库和 example
2. `icoo` 主链和 CLI 能作为独立产品口径交付
3. `examples/proxy` 能作为独立可用样例口径交付
4. `examples/proxy/smoke.ic` 稳定通过
5. `GOFLAGS=-p=1; go test ./...` 稳定通过
6. `docs/mvp-status.md`、`docs/api.md`、`docs/proxy_refactor_feedback.md`、本文档彼此不冲突
7. 没有明显阻断使用的已知问题仍处于未收口状态
8. 新增工作已从“继续造能力”切换为“围绕现有能力补稳、补文档、补测试”

## 十、建议的发版后方向

`v0.1` 发出后，再按下面顺序继续演进更稳妥：

1. 继续把 `examples/proxy` 当作服务端回归样例
2. 沿 `std.http.server` / `std.express` 补更底层、更代理友好的 HTTP 能力
3. 继续观察 `std.service` 是否需要事件/日志/脱敏辅助
4. 补 parser / sema / compiler 白盒测试
5. 再判断是否值得做对象展开、可空访问等语法增强

这里的关键原则是：

> 先把当前这轮由真实重构倒逼出来的能力交付扎实，再决定下一轮扩张，而不是把交付窗口再次变成实验窗口。
