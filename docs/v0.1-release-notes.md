# Icoo v0.1 发布说明

`v0.1` 是 Icoo 第一个以“可交付、可说明、可回归”为目标收口的版本。

它不是“第一个能跑的版本”，而是第一个明确从持续开发状态切换到版本交付状态的版本。

相关文档：

- 当前状态：`docs/mvp-status.md`
- v0.1 交付规划：`docs/v0.1-delivery-plan.md`
- Runtime API：`docs/api.md`
- proxy 重构反馈：`docs/proxy_refactor_feedback.md`

## 版本定位

`v0.1` 的核心不是继续扩张特性，而是把已经形成的能力边界稳定下来：

- 主链稳定
- CLI 可用
- API 可嵌入
- examples 可回归
- 服务端样例可验证

这一版特别强调两类交付结果：

1. `icoo` 本体交付
2. `examples/proxy` 可用交付

## 本版重点

### 1. 语言与工具链已经超出 MVP

当前版本已经具备：

- `check` / `run`
- `repl`
- `bundle` / `build` / `extract` / `inspect`
- 完整的源码执行主链
- 文件模块与标准库模块系统
- `pkg/api` 运行时门面

### 2. 语言能力范围已经形成

当前已纳入可交付范围的主要能力包括：

- 函数、闭包、数组、对象、控制流
- import/export
- class / inheritance / decorator
- throw / try / catch / finally / `expr?`
- chan / go / select
- type / interface / satisfies

### 3. 标准库补齐了一批服务端薄能力

围绕 `examples/proxy` 这轮重构反馈，本版补齐并固化了以下能力：

- `std.object`
- `req.header(name)` / `req.hasHeader(name)` / `req.json`
- 双参 handler `fn(req, res)`
- `res.status(...)` / `res.statusCode()` / `res.setHeader(...)` / `res.write(...)` / `res.json(...)` / `res.end(...)`
- `res.proxy(req, options)`
- `req.requestId`
- `std.observe.recent(limit)`
- `std.service.create(...)`
- `std.orm.model(...)`

### 4. proxy 已经成为服务端回归样例

`examples/proxy` 在 `v0.1` 里不再只是普通 example，而是：

- 服务端能力样例
- 标准库设计反馈样例
- proxy 重构落地样例
- 发版前的关键回归样例

## 推荐验证方式

在当前 Windows 环境下，推荐优先使用低并发验证：

```powershell
$env:GOFLAGS='-p=1'
go test ./...
```

proxy 相关重点验证：

```powershell
$env:GOFLAGS='-p=1'
go test ./pkg/api -run 'StdService|ExamplesRun/smoke' -v
go run ./cmd/icoo run examples/proxy/smoke.ic
```

## 当前适合做什么

`v0.1` 当前适合：

- 作为语言实现与 VM/CLI 工具链基础版本交付
- 作为 embedding/runtime 实验基础
- 作为标准库和 examples 回归基线
- 作为轻量服务脚本和服务样例验证平台

## 当前还不适合做什么

`v0.1` 当前还不应被误解为：

- 生产级网关产品
- 重量级服务框架
- 高性能流式协议桥接平台
- 完整监控/追踪系统
- 大规模宿主集成 SDK

这些复杂度仍应优先由 Go 宿主层承接。

## 版本收口原则

`v0.1` 之后继续演进时，优先顺序应保持为：

1. 稳定现有能力
2. 补回归测试
3. 补文档
4. 再决定是否扩张能力面

关键原则：

> 先把真实样例倒逼出来的能力交付扎实，再决定下一轮扩张，而不是把每次发版窗口都重新变成实验窗口。
