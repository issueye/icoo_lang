# Icoo v0.1.1 开发计划

本文档基于本地代码对比：

- `proxy/icoo_server`
- `examples/proxy`

目标不是追求两者“字面等价”，而是判断当前这轮重构反哺已经完成到什么程度，以及 `v0.1.1` 应该优先补哪一层能力。

## 当前落地状态（2026-05-02）

`v0.1.1` 已完成 Phase 1 的第一步落地：

- `examples/proxy` 已支持 `OpenAI Chat <-> OpenAI Responses` 的非流式 JSON 双向转换
- `examples/proxy/smoke.ic`
- `pkg/api` 的 `ExamplesRun/smoke`
- `GOFLAGS=-p=1 go test ./...`

均已在本地通过。

本轮当时仍然明确未覆盖：

- Anthropic 相关协议转换
- SSE / stream 翻译与聚合
- 持久化 history
- admin 操作面增强

因此 `v0.1.1` 目前的实际状态应理解为：

- 已从“协议内转发器”进入“基础跨协议代理”
- 但仍停留在非流式、双协议、样例级落地阶段

同日继续完成了 Phase 2 的最小闭环：

- 标准库补齐了 HTTP/Express 响应侧的 SSE 写出 helper
- `std.net.sse.client` 补齐了一次性请求并收集事件的入口
- `examples/proxy` 已能把上游 `OpenAI Responses` SSE 响应聚合为 JSON，再继续走既有的协议翻译链路

这一步仍然不等于“完整流式翻译已完成”，但已经证明语言侧具备了：

- 稳定写 SSE
- 稳定收 SSE
- 稳定把 Responses 流收敛回 JSON

截至 `2026-05-03`，Phase 2 又继续向前推进了一步：

- `examples/proxy` 已支持 `OpenAI Responses SSE -> OpenAI Chat SSE` 的最小逐事件翻译
- 当前已打通 assistant role chunk、text delta chunk、tool call start / arguments delta、finish chunk、`[DONE]` 收尾
- proxy smoke 已新增真实下游 SSE 消费断言

同日还补上了 Phase 1 里最值得优先完成的第三协议主路径：

- `Anthropic Messages -> OpenAI Responses` 最小 JSON 请求/响应转换
- `OpenAI Responses -> Anthropic Messages` 最小 JSON 请求/响应转换
- `/v1/messages` 与 `/v1/responses` 已分别新增跨协议 alias 验证路径
- proxy smoke 已新增双向跨协议断言，覆盖 `system` / text / `max_tokens` / usage

## 一、结论先行

从本地代码对比看，`examples/proxy` 对 `proxy/icoo_server` 的重构承接已经完成了“可运行代理样例”这一层，但还没有完成“协议转换服务”和“完整服务工程化外壳”这一层。

更直白地说：

- `v0.1` 已经证明 `icoo` 能做轻量代理和服务端编排
- `v0.1` 还没有把 Go 版 proxy 的核心工程能力完整回流到语言侧

如果把 `proxy/icoo_server` 视为对照基线，那么当前完成度可以分两层理解：

### 1. 以“基础代理样例”衡量

完成度大约在 `70%~80%`。

已经具备：

- 路由选择
- 鉴权
- model alias 改写
- upstream path 改写
- request id 透传
- recent requests / counters / latency
- health / ready / admin monitor
- 可运行冒烟回归

### 2. 以“Go 版完整服务能力”衡量

完成度更接近 `35%~45%`。

仍然缺少：

- 跨协议请求/响应转换
- 流式协议转换与聚合
- 持久化 traffic recorder
- 完整 admin CRUD 控制面
- supplier health 与运行时控制
- 链路日志、脱敏与请求镜像

这里的百分比是基于代码结构和能力覆盖面的工程判断，不是精确统计值。

## 二、完成度对比

### A. 已基本完成的回流项

这些能力已经不再只是 Go 版独有，而是明确进入了语言和标准库：

1. 安全对象访问与浅合并
   - `std.object`
2. 服务端请求读取能力
   - `req.header(name)` / `req.hasHeader(name)` / `req.json`
3. 直接响应控制
   - 双参 handler `fn(req, res)`
   - `res.status(...)` / `res.setHeader(...)` / `res.write(...)` / `res.json(...)` / `res.end(...)`
4. 成功路径直通代理
   - `res.proxy(req, options)`
5. 请求链路标识
   - `req.requestId`
6. 轻量观测
   - `std.observe.recent(limit)`
   - `std.service.create(...)`
7. 薄 ORM
   - `std.orm.model(...)`

这些项支撑了 `.ic` 版本当前的主路径可用性。

### B. 部分完成但仍偏薄的项

这些能力在 `.ic` 侧已经有影子，但和 Go 版相比还停在“样例级”：

1. 管理接口
   - `.ic` 版已有 `/healthz`、`/readyz`、`/admin/routes`、`/admin/requests`、`/admin/monitor`
   - Go 版已有完整 `/api/*` 控制面
2. 监控
   - `.ic` 版已有 recent requests、counters、latency
   - Go 版还有持久化 traffic、分页查询、协议过滤和统计视图
3. 配置
   - `.ic` 版已有默认配置 + env 覆盖 + JSON 文件覆盖
   - Go 版已经走向 project settings / ui prefs / reload
4. 健康
   - `.ic` 版已有 service health / ready
   - Go 版还有 supplier health check

### C. 仍明显缺失的项

这些是当前 `examples/proxy` 与 Go 版差距最大的部分。

#### 1. 协议转换层

Go 版已有：

- `translation.ConvertRequest(...)`
- `translation.ConvertResponse(...)`
- OpenAI Chat / OpenAI Responses / Anthropic 三套模型间的双向转换

`.ic` 版当前只有：

- `json_helpers.rewriteBody(...)`
- 本质上只是改 `model`

也就是说，`.ic` 版现在还不是“协议网关”，只是“协议内轻量转发器”。

#### 2. 流式转换层

Go 版已有：

- SSE 读取与事件解析
- `TranslateResponsesStreamToAnthropic(...)`
- `TranslateResponsesStreamToChat(...)`
- `TranslateAnthropicStreamToChat(...)`
- `AggregateResponsesStreamToJSON(...)`

`.ic` 版当前只有：

- `res.proxy(...)` 的成功路径直通

也就是说，`.ic` 版能“透传”，但还不能“理解并翻译流”。

#### 3. 持久化 traffic recorder

Go 版已有：

- `internal/traffic/service.go`
- LevelDB 持久化请求记录
- 分页、过滤、协议选项、token stats、平均延迟

`.ic` 版当前只有：

- 内存 recent requests
- 内存 counters / latency

这意味着当前 `.ic` 版监控更接近 runtime observe，而不是完整 traffic history。

#### 4. 完整 admin 控制面

Go 版已有：

- suppliers / endpoints / auth-keys / route-policies / model-aliases / settings / ui-prefs / traffic

`.ic` 版当前只有：

- read-only 管理接口
- 没有 CRUD 控制面
- 没有 reload / settings / supplier management

#### 5. 链路日志与请求镜像

Go 版已有：

- chain logger
- body redaction
- downstream / upstream request/response 保存

`.ic` 版当前只有：

- recent request 记录

所以 `.ic` 版现在可以做“结果监控”，还做不到“完整链路审计”。

## 三、为什么这些差距不能一次性全补完

`v0.1.1` 不应该试图把 Go 版 proxy 全量复制回语言侧。

原因很明确：

1. 有些复杂度天然更适合 Go 承接
   - 高强度流式处理
   - 大量 typed model 映射
   - 成熟持久化依赖
2. 有些差距并不说明语言表达力不够
   - 而是标准库还缺一层薄能力
3. `v0.1.1` 的正确目标不是“追平 Go”
   - 而是继续提炼哪些复杂度值得回流，哪些仍应留在宿主层

## 四、v0.1.1 的版本目标

`v0.1.1` 建议定位为：

> 从“可运行代理样例”推进到“具备基础协议转换能力的代理样例”，同时继续薄化标准库，而不是直接走向完整产品化控制面。

核心目标分三层：

### 目标 1：让 `.ic` proxy 正式进入“协议转换代理”阶段

至少补齐：

- OpenAI Chat -> OpenAI Responses
- OpenAI Responses -> OpenAI Chat
- Anthropic -> OpenAI Responses
- OpenAI Responses -> Anthropic

先做非流式 JSON 路径。

### 目标 2：补最小流式基础件

不是一上来做全量流式翻译，而是先补：

- SSE 事件读取基础
- SSE 事件写出基础
- Responses 流聚合为 JSON

先让语言侧具备“看懂流”和“聚合流”的底座。

### 目标 3：补服务控制面的最小闭环

先不要复制 Go 版整个 `/api/*`，而是优先补：

- supplier health
- 简单 runtime reload
- 请求历史的可选持久化

## 五、v0.1.1 开发主题

### Theme A：协议转换薄层

这是 `v0.1.1` 的最高优先级。

建议输出物：

1. `examples/proxy/lib/translate.ic`
2. 一组标准库或 helper 级 JSON 转换原语
3. proxy smoke 的跨协议用例

建议边界：

- 先做 request/response JSON 转换
- 先不做全量 reasoning / tool / thinking block 的完美映射
- 先把主路径打通，再补字段细节

### Theme B：SSE / stream 基础原语

建议优先补到底层标准库，而不是直接把复杂逻辑塞进 example。

建议能力：

1. `std.net.http.client` 增加流式读取入口
2. `std.net.http.server` / `std.express` 增加 SSE 写出辅助
3. 事件流解析 helper
4. Responses stream -> JSON aggregate helper

这层完成后，`.ic` proxy 才有条件继续做流式协议转换。

### Theme C：可选持久化 traffic

这里不要直接照搬 Go 版 LevelDB 设计。

更适合 `icoo_lang` 的做法是提供薄接口，例如：

- 基于 `std.db` / `std.orm` 的请求历史记录
- 可选文件落盘
- 分页和简单过滤

先把“历史可查”补出来，不急着追求完整运维面。

### Theme D：服务控制面的最小增强

建议只补最有价值的三项：

1. `/admin/suppliers/health`
2. `/admin/reload`
3. `/admin/history`

不要在 `v0.1.1` 里直接追求 suppliers / endpoints / ui-prefs 全 CRUD 控制台。

### Theme E：链路日志与脱敏辅助

这是很值得标准库化的一层，因为 Go 版已经说明它在真实服务里是刚需。

建议能力：

- 结构化日志 helper
- header/body 脱敏 helper
- request/response snapshot helper

这层先做 helper，不做完整 logging framework。

## 六、v0.1.1 的分阶段计划

### Phase 1：JSON 协议转换

目标：

- `.ic` proxy 可以跨协议处理非流式请求/响应

范围：

- Chat <-> Responses
- Responses <-> Anthropic
- 只做非流式 JSON

验收：

- 新增 smoke / module tests
- 至少 4 条跨协议路径跑通

当前状态：

- `Chat -> Responses` 已完成
- `Responses -> Chat` 已完成
- `Anthropic -> Responses` 已完成最小 JSON 主路径
- `Responses -> Anthropic` 已完成最小 JSON 主路径
- 已新增并跑通 4 条非流式跨协议 smoke 路径

这一步反哺出的语言/标准库结论比较明确：

- 现有 `req.json` / `server.forward(...)` / `res.proxy(...)` 已足以支撑非流式协议转换
- `std.object` 与 `std.json` 的薄能力组合，可以承接大部分 JSON 形状映射
- 真正缺的不是专用语法，而是更稳定的流式读写与事件聚合原语

这一轮继续反哺出的结论也更具体了：

- 三协议最小 JSON 互转并不要求语言引入专用协议语法
- 真正需要沉淀的是稳定的 JSON shape helper 与更薄的跨协议转换习惯用法
- Anthropic 的复杂 tool / thinking / stream 细节，应该在主路径打通后再逐层吸收，避免一次性把 example 变成庞大的协议兼容层

### Phase 2：SSE 基础件

目标：

- 语言侧具备事件流读取、写出和聚合基础

范围：

- SSE parse
- SSE write
- Responses stream aggregate

验收：

- 有单独的 stdlib 测试
- proxy example 可消费并聚合上游 stream

当前状态：

- `response.sse(...)` / `express.response.sse(...)` 已落地
- `std.net.sse.client.request(...)` 已落地
- `std.net.sse.client.connect(...)` 已支持带 `method/headers/body` 的上游流式请求
- proxy smoke 已新增“上游强制流式 -> 聚合 -> 协议翻译”路径并跑通
- proxy smoke 已新增“上游 Responses SSE -> 下游 Chat SSE”路径并跑通
- `ExamplesRun/19_sse_stream`
- `ExamplesRun/smoke`
- `GOFLAGS=-p=1 go test ./...`

均已在本地通过。

这一阶段反哺出的设计结论也更明确了：

- 对语言侧最有价值的不是一上来做完整流式协议翻译
- 而是先把 `SSE write`、`SSE collect`、`stream -> JSON aggregate` 这三块底座薄化到标准库
- 在这三块稳定后，再继续补 `Responses -> Chat stream` 和 `Responses -> Anthropic stream` 会更自然

当前 Phase 2 的实际边界应理解为：

- `Responses -> Chat` 已有最小文本流翻译路径
- `Responses -> Chat` 已有最小 tool call 流翻译路径
- 还没有补 reasoning block / Anthropic stream 翻译
- 也还没有进入完整的多协议流式事件桥接阶段

同日也开始补上了 Phase 3 的最小可交付闭环：

- `examples/proxy` 已接入基于 `std.db + std.orm` 的 SQLite history store
- `/admin/history` 已支持最小查询与跨重启读取
- `/admin/history/clear` 已支持清理
- `/admin/suppliers/health` 已可读取当前 upstream 健康状态
- 为了支撑 admin query 参数，标准库同步补上了 `std.math.parseInt(...)`

### Phase 3：最小持久化 history

目标：

- 从 recent requests 走向可查询 history

范围：

- request history 持久化
- page / limit
- 简单 protocol / status filter

验收：

- `/admin/history` 可读
- 重启后历史仍可查

当前状态：

- `/admin/history` 已完成
- history 已通过 proxy smoke 验证跨重启保留
- 当前仍未补分页游标、复杂过滤与聚合统计视图

### Phase 4：最小控制面增强

目标：

- 让 proxy 从“观测面”走向“最小操作面”

范围：

- health check
- reload
- history clear / query

验收：

- 管理接口可直接驱动这三项操作

## 七、v0.1.1 不建议纳入的内容

为了防止失控，建议明确排除：

1. 全量复刻 Go 版 admin API
2. 完整 UI prefs / settings 后台
3. 全协议、全字段、全事件的流式翻译一次补齐
4. 重量级 tracing / metrics 系统
5. 为了贴近 Go 版而引入太多只服务单个 example 的专用语法

## 八、建议的完成度目标

如果 `v0.1.1` 按上面范围推进，建议目标不是“和 Go 版完全对齐”，而是把整体完成度推进到：

- 基础代理样例视角：`85%+`
- 服务工程能力视角：`55%~65%`

这里的关键不是数字，而是能力层级变化：

- 从“协议内转发器”进入“基础协议转换代理”
- 从“纯内存观察”进入“可查询 history”
- 从“只读监控面”进入“最小控制面”

## 九、v0.1.1 验收标准

当下面条件同时成立时，可以认为 `v0.1.1` 达标：

1. `.ic` proxy 至少支持一组三协议稳定的跨协议 JSON 转换路径
2. 标准库具备最小 SSE 基础原语
3. proxy 具备最小持久化 history 能力
4. proxy 具备最小 reload / supplier health 能力
5. 新增能力均有 smoke 和 `pkg/api` 测试覆盖
6. 没有为了追平 Go 版而让标准库变成过度专用实现

## 十、最终建议

`v0.1.1` 最值得做的，不是“继续让 example 更大”，而是继续把 Go 版里那些已经证明有价值、但仍能以薄层形式抽象出来的能力回流到语言侧。

最推荐的顺序是：

1. JSON 协议转换
2. SSE 基础原语
3. 持久化 history
4. 最小控制面
5. 日志 / 脱敏辅助

这条顺序能最大化保证：

- proxy 的重构继续落地
- 语言能力继续变强
- 标准库不会因为一次版本追赶而失控膨胀
