# Icoo v0.2.0 交付规划

本文档的目标不是继续描述“还能做什么”，而是把 `examples/proxy` 推进到一个能够作为 `proxy/icoo_server` 的**功能一致重构版本**交付的状态。

这里的“功能一致”有明确边界：

- 指下游可见能力、管理能力、协议能力和运维能力达到同一交付层级
- 不要求 `.ic` 版本在目录结构、类型建模或实现细节上与 Go 版字面一致
- 允许仍由 Go 宿主层承接高性能底层实现，但 `.ic` 版本必须能稳定提供同等级的功能闭环

换句话说，`v0.2.0` 的目标不是“把 Go 版抄回来”，而是：

> 让 `icoo_lang` 交付一个在功能面、控制面、协议面和验收面上都足够完整的 proxy 重构版本。

## 一、v0.2.0 的版本定义

`v0.2.0` 应定位为：

- 第一个可交付的“功能一致重构版 proxy”
- 第一个把协议转换、流式桥接、持久化 history、最小控制面和链路观测统一起来的版本
- 第一个能用 `.ic` proxy 与 `proxy/icoo_server` 做能力面对照，而不是只能做样例对照的版本

这意味着 `v0.2.0` 的交付结果仍然有两个：

1. `icoo_lang` 本体继续增强并稳定
2. `examples/proxy` 达到可作为重构交付物验收的完整度

## 二、当前基线（2026-05-03）

当前 `examples/proxy` 已具备：

- `OpenAI Chat <-> OpenAI Responses` 非流式 JSON 双向转换
- `Anthropic Messages <-> OpenAI Responses` 最小 JSON 主路径
- `OpenAI Responses -> OpenAI Chat` 最小 SSE 直译
- SSE 聚合回 JSON
- 最近请求、计数器、latency
- SQLite 持久化 history
- `/admin/history`、`/admin/history/clear`、`/admin/suppliers/health`
- 基础 `healthz` / `readyz` / `routes` / `monitor`

当前仍未达到“功能一致交付”的关键缺口：

1. 流式协议转换还不完整
2. 管理控制面只有最小只读接口
3. 目录模型和策略模型还没有完整持久化
4. 请求镜像、链路日志、脱敏和 traffic 统计仍偏薄
5. runtime reload 与 supplier health check 还不完整
6. 复杂协议字段的一致性仍不够

## 三、v0.2.0 的完成定义

当以下条件同时成立时，才能认为 `v0.2.0` 达到“功能一致重构版”：

1. `.ic` proxy 覆盖三协议主要 JSON 路径与主要流式路径
2. `.ic` proxy 具备最小但完整的 `/api/*` 管理控制面
3. routes / suppliers / endpoints / aliases / auth keys / policies 能持久化并可通过接口管理
4. traffic history 不只是“请求列表”，而是可分页、可过滤、可统计
5. request / response mirror、链路日志、脱敏辅助具备可用闭环
6. reload、supplier health、model cache 至少具备最小可操作版本
7. `examples/proxy/smoke.ic` 与 `pkg/api` 回归能够证明这些能力

## 四、与 Go 版的差距分层

### A. 已基本追平的层

- 基础代理路由
- model alias 主路径
- 基础鉴权
- request id
- 基础 health / ready / monitor
- 最小 history
- 三协议最小 JSON 互转

### B. 部分追平但仍偏薄的层

- 流式转换
- supplier health
- 持久化 history
- admin 查询面
- 服务监控骨架

### C. 仍明显缺失的层

#### 1. 完整控制面

Go 版已有：

- `/api/suppliers`
- `/api/endpoints`
- `/api/auth-keys`
- `/api/route-policies`
- `/api/model-aliases`
- `/api/settings`
- `/api/ui-prefs`
- `/api/traffic`
- `/api/proxy/reload`

`.ic` 版当前缺：

- CRUD 写接口
- page / all 查询模式
- secret reveal
- reload
- settings / ui prefs

#### 2. 完整流式桥接

Go 版已有：

- `Responses -> Anthropic stream`
- `Responses -> Chat stream`
- `Anthropic -> Chat stream`
- `Responses stream -> JSON aggregate`

`.ic` 版当前缺：

- `Responses -> Anthropic` SSE 翻译
- `Anthropic -> Chat` SSE 翻译
- reasoning / thinking / tool_use 的更完整事件翻译

#### 3. 完整 traffic 服务

Go 版已有：

- 分页 traffic
- protocol filter
- token stats
- average latency
- clear
- protocol options

`.ic` 版当前缺：

- 独立 traffic 视图
- 聚合统计视图
- 更丰富的过滤项
- 与 request mirror 的联动

#### 4. 目录与策略服务

Go 版已有：

- suppliers
- endpoints
- model aliases
- route policies
- catalog resolver
- supplier model cache

`.ic` 版当前缺：

- 持久化目录模型
- 策略配置管理
- 运行时路由重建
- supplier model cache

#### 5. 链路审计

Go 版已有：

- downstream request/response save
- upstream request/response save
- body redaction
- chain logger

`.ic` 版当前缺：

- 持久化 request/response mirror
- 通用脱敏 helper
- 可配置的链路日志输出

## 五、v0.2.0 核心目标

### 目标 1：完成“主要功能面一致”

至少补齐：

- 三协议 JSON 主路径字段一致性
- 三协议主要流式路径
- 上游错误透传和统一错误格式
- traffic / mirror / usage 统计主路径

### 目标 2：完成“最小控制面一致”

至少补齐：

- suppliers
- endpoints
- auth keys
- route policies
- model aliases
- settings
- ui prefs
- reload

### 目标 3：完成“最小运维面一致”

至少补齐：

- persisted history / traffic
- supplier health check
- request/response mirror
- structured logs + redaction

## 六、分阶段计划

### Phase 0：冻结交付契约

目标：

- 先冻结 `v0.2.0` 对“功能一致”的定义
- 明确哪些能力必须交付，哪些能力不纳入

输出物：

- 本文档
- 一份按功能域的 gap checklist

验收：

- 后续开发不再以“是否更像 Go”判断完成，而以“是否满足功能契约”判断

### Phase 1：持久化目录模型与服务层重构

目标：

- 让 `examples/proxy` 从硬编码 config/alias 进入“持久化目录 + service 层”阶段

范围：

- SQLite 持久化 `suppliers`
- SQLite 持久化 `endpoints`
- SQLite 持久化 `auth_keys`
- SQLite 持久化 `model_aliases`
- SQLite 持久化 `route_policies`
- service 层封装 CRUD 与校验

建议输出物：

- `examples/proxy/lib/services/`
- `examples/proxy/lib/store/`
- 路由解析从 `catalog.ic` 迁到更稳定的 catalog service

验收：

- proxy 启动可从持久化目录加载路由
- smoke 可覆盖目录读取和 alias 解析
- 管理接口可读出持久化模型

### Phase 2：最小完整 admin 控制面

目标：

- 从 `/admin/*` 只读面推进到 `/api/*` 最小可操作控制面

范围：

- `/api/overview`
- `/api/state`
- `/api/proxy/reload`
- `/api/suppliers`
- `/api/endpoints`
- `/api/auth-keys`
- `/api/route-policies`
- `/api/model-aliases`
- `/api/settings`
- `/api/ui-prefs`
- `/api/traffic`

边界：

- 不要求先做 UI
- 先完成 HTTP API

验收：

- 每组资源都有 list/page/upsert/delete 最小组合
- reload 可触发目录重建
- smoke 或单测覆盖至少一条 CRUD 闭环

### Phase 3：协议转换字段完整度提升

目标：

- 把当前“最小主路径可用”推进到“主要字段一致”

范围：

- Anthropic `tool_use` / `tool_result`
- OpenAI Chat `tool_calls`
- Responses `function_call` / `function_call_output`
- usage / finish reason / stop reason
- reasoning / thinking 主字段映射

边界：

- 先追主要字段一致
- 不追求一轮补全所有边角字段

验收：

- 新增 translation module tests
- 至少覆盖 tool / reasoning / usage 的双向断言

### Phase 4：完整流式桥接

目标：

- 让 `.ic` proxy 进入“主要流式能力一致”阶段

范围：

- `Responses -> Anthropic` SSE
- `Anthropic -> Chat` SSE
- `Responses -> Chat` reasoning/tool 边角补全
- 流式错误事件统一
- usage 流式采集稳定化

边界：

- 先围绕已有三协议主路径
- 不做泛化到任意协议桥的抽象过度设计

验收：

- smoke 增加三协议流式断言
- `pkg/api` 增加 stream example tests

### Phase 5：traffic、mirror 与脱敏闭环

目标：

- 把 history 从“请求列表”推进到“可运维的 traffic 视图”

范围：

- request/response mirror 持久化
- body/header redaction helper
- `/api/traffic` 分页、过滤、token stats、平均延迟
- clear traffic
- protocol options

反哺语言点：

- `std.service` 事件/日志能力继续薄化
- `std.object` / `std.json` / `std.fs` 在 request snapshot 场景下继续打磨

验收：

- traffic 接口与 mirror 数据可联动验证
- 脱敏字段不泄漏 secret / auth header

### Phase 6：health、reload 与 cache 收口

目标：

- 把运行时控制面补齐到可交付层级

范围：

- supplier health check
- runtime reload
- supplier model cache
- reload 后路由生效验证

验收：

- health check 可针对指定 supplier 执行
- reload 能反映目录变更
- cache 行为有测试保护

### Phase 7：交付收尾与稳定性

目标：

- 将 `v0.2.0` 从“功能已经有”收口到“版本可以交付”

范围：

- docs
- smoke
- regression
- release notes
- acceptance checklist

验收：

- `GOFLAGS=-p=1 go test ./...`
- proxy smoke
- admin API smoke
- stream smoke

全部稳定通过

## 七、建议的实现顺序

推荐顺序如下：

1. 持久化目录模型
2. admin 控制面
3. 协议字段完整度
4. 完整流式桥接
5. traffic / mirror / redaction
6. reload / health / cache
7. 文档与发布收尾

原因：

- 目录模型和控制面是其余能力的承载面
- 流式桥接应建立在稳定目录和管理能力之上
- traffic / audit 需要在主要协议路径稳定后补，避免重复返工

## 八、v0.2.0 不建议纳入的内容

为了确保版本能交付，以下内容不建议并入 `v0.2.0`：

1. 全量复刻 Go 版目录结构
2. 引入重量级 tracing / metrics 系统
3. 为 proxy 单独设计过多专用语法
4. 一次性做成完整前端控制台
5. 为了“功能一致”而牺牲标准库薄层原则

## 九、标准库反哺清单

`v0.2.0` 期间，最值得继续沉淀到语言/标准库的能力不是“更多协议专用语法”，而是：

1. 更稳定的流式 HTTP/SSE 原语
2. 更顺手的 query / pagination helper
3. 更薄的 service/log/redaction helper
4. 更完整的 `std.db + std.orm` 服务端 CRUD 习惯用法
5. 更稳定的 request/response snapshot helper

## 十、建议的完成度目标

如果 `v0.2.0` 按上述范围推进，建议目标是：

- 基础代理样例视角：`95%+`
- Go 版服务能力视角：`80%~90%`

这里关键不是数字，而是层级变化：

- 从“可运行样例”进入“功能一致交付物”
- 从“最小观测面”进入“最小控制面 + 运维面”
- 从“主路径协议转换”进入“主要功能面协议一致”

## 十一、v0.2.0 验收标准

当下面条件同时成立时，可以认为 `v0.2.0` 达标：

1. `.ic` proxy 能以独立交付物身份覆盖主要 proxy 功能面
2. `/api/*` 管理控制面具备最小完整度
3. 三协议主要 JSON 与流式路径均有稳定回归
4. traffic / history / mirror / redaction 形成闭环
5. reload / supplier health / cache 可用
6. 版本文档、README、release notes、acceptance checklist 完整

## 十二、最终建议

`v0.2.0` 不应再把重心放在“单点补 feature”，而应正式转入：

> 围绕 `examples/proxy` 建立一个可以对照 `proxy/icoo_server` 验收的完整重构交付版本。

这会直接带来两类收益：

1. `icoo_lang` 会因为真实服务复杂度而继续获得正确的标准库反馈
2. `proxy` 重构终于不再只是“样例持续逼近”，而是可以形成一次正式版本交付
