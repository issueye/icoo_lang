# Icoo Proxy v0.1 可用交付说明

本文档定义 `examples/proxy` 在 `v0.1` 阶段的交付口径。

它的目标不是把这个目录包装成完整产品，而是明确：

- 这个 proxy 已经能做什么
- 如何运行与验证
- 它为什么能作为 `v0.1` 的可用交付
- 它的边界在哪里

相关文档：

- 总交付规划：`docs/v0.1-delivery-plan.md`
- proxy 运行说明：`examples/proxy/README.md`
- proxy 重构反馈：`docs/proxy_refactor_feedback.md`

## 交付定位

`examples/proxy` 在 `v0.1` 中承担的是“可用服务样例交付”，不是“生产网关交付”。

它的价值在于：

1. 验证 `icoo` 当前已经能支撑一个可运行的代理服务
2. 验证这轮从 `proxy/icoo_server` 反哺出来的标准库和运行时能力已经真正落地
3. 为后续服务端能力演进提供稳定回归样例

## 当前能力

当前 proxy 已具备以下可验证能力：

- 路由匹配与上游路径改写
- 模型别名改写
- Bearer 鉴权
- 请求 ID 透传
- 直接代理响应回写
- 健康检查和就绪检查
- recent requests 记录
- 服务级 counters
- 轻量 latency summary
- 管理接口输出监控快照

## 运行方式

### 持续运行

```powershell
go run ./cmd/icoo run examples/proxy/app.ic
```

### 冒烟验收

```powershell
go run ./cmd/icoo run examples/proxy/smoke.ic
```

这个脚本会自动启动临时 upstream 和 proxy，并校验关键行为。

## 验收接口

运行中的 proxy 当前暴露：

- `/healthz`
- `/readyz`
- `/admin/routes`
- `/admin/models`
- `/admin/requests`
- `/admin/monitor`

其中：

- `/admin/requests` 关注 recent requests、counters 和 latency
- `/admin/monitor` 关注完整服务监控快照

## v0.1 验收标准

以下条件同时成立时，视为 proxy 达到 `v0.1` 可用交付标准：

1. `go run ./cmd/icoo run examples/proxy/smoke.ic` 通过
2. 成功请求可完成鉴权、路由改写、上游转发和响应回写
3. 未授权请求可返回 401
4. route not found 可返回 404
5. 请求 ID 在下游响应和上游透传中保持一致
6. `/healthz`、`/readyz`、`/admin/requests`、`/admin/monitor` 可读
7. recent requests、counters、latency 数据可读
8. 主路径已不再依赖旧 store 监控实现

## 当前依赖的语言能力

这个 proxy 当前直接依赖本轮回流出来的能力：

- `std.object`
- `std.http.server`
- `std.express`
- `std.observe`
- `std.service`
- `std.orm`
- `req.requestId`
- `res.proxy(req, options)`

因此它不是“无关痛痒的 demo”，而是 `v0.1` 服务端能力是否落地的核心回归面。

## 当前边界

这个 proxy 仍然不是：

- 完整产品化 API 网关
- 流式协议转换平台
- 全量流量镜像与持久化系统
- 生产级监控平台

这些复杂度仍然更适合由 Go 宿主层承接。
